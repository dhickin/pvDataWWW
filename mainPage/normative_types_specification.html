Status of this Document

This is the 10 November 2011 version of the EPICS v4 normative types specification. This is an editors draft, composed soley of the type definitions agreed at the Oct 2011 PSI EPICS V4 workshop.

Since this is an editors draft, it is not intended for comments.

The terms MUST, MUST NOT, SHOULD, SHOULD NOT, REQUIRED, and MAY when highlighted (through style sheets, and in uppercase in the source) are used in accordance with RFC 2119 [RFC2119]. The term NOT REQUIRED (not defined in RFC 2119) indicates exemption.

Introduction

EPICS version 4 supplies basic types types. Additionally, EPICS V4 defines a set of Normative types that build on the PVStructure type.

The Normative Types

NOTE: There is no specific support for unsigned types. * See definition in RESOLUTION of 4-Oct.

Atomic Control types typeIdentifier -------------- typeIdentifier is a long whose bitfield value is used to uniquely identify a PVStructure instance as an example of a data type. The first 16 bits (of 64) are reserved to identify the normative types. long typeIdentifier_t The following control types are defined as structures of atomic types, and encoded as PVStructure. structure enum_t int index string[] choices structure timeStamp_t long secsPastEpoch int nanoseconds int userTag structure alarm_t int severity* int status* string message structure nameValuePair_t string name string value *severity is defined as an int (not an enum_t), but should be functionally interpreted only as the enumeration {NONE, MINOR, MAJOR, INVALID, UNDEFINED } *status is defined as an int (not an enum_t), but should be functionally interpreted only as the enumeration {NONE, DEVICE, DRIVER, RECORD, DB, CONF, UNDEFINED, CLIENT } Array of atomic type: float[] (IEEE 754) double[] (IEEE 754) short[] (16 bit) int[] (32 bit) long[] (64 bit) byte[] (8 bit) string[] boolean[] Array of Control Types: enum_t[] timeStamp_t[] alarm_t[] nameValuePair_t[] Control General-Normative Types ------------------------------- The set of control general-normative types is the composed of each of the general-normative types, plus timestamp and severity. These are encoded as PVStructure. Syntax of type definition structure type-name type-name field-name type-name field-name type-name identifies the type - one of those above. field-name identifies the field in the introspection interface. NTFloat ------- Defined as: structure NTFloat alarm_t alarm timeStamp_t timeStamp float value A decoder would interpret this as, if you are given structure which is named NTFloat, you would expect to find that this structure is defined as an alarm_t field whose introspection name is "alarm", followed by a timeStamp_t type field, whose introspection name is "timeStamp", followed by a float, whose introspection name is "value". NTDouble NTShort (16 bit) NTInt (32 bit) NTLong (64 bit) NTByte (8 bit) NTString (see resolution 4-Oct-2011 meeting) NTBoolean NTEnum ------ Defined as: structure NTEnum typeIdentifier_t typeIdentifier alarm_t alarm timeStamp_t timeStamp enum_t value Array of general-normative types. The array as a whole has NTTimestamp and NTAlarm [still have to argue about that]. NTFloatArray NTDoubleArray NTShortArray (16 bit) NTIntArray (32 bit) NTLongArray (64 bit) NTByteArray (8 bit) NTStringArray (see resolution 4-Oct-2011 meeting) NTBooleanArray NTEnumArray Specific Normative Types ======================== Specific normative types are types defined by the EPICS V4 working group, that are oriented towards specific uses. These are also encoded as PVStructure. Times always in seconds. Frequencies always in Hz. NTMultichannelArray ------------------- MultichannelArray is a type to aggregate an array of scalar values into a single array variable. structure NTMultichannelArray typeIdentifier_t typeIdentifier alarm_t alarm timeStamp_t timeStamp double[] values int[] severities double[] positions // The real absolute or relative position of each scalar. If not relevant, should be valued 0.0. double[] deltaTimes // The time relative to the timeStamp in seconds. If not relevant, should be valued 0.0 NTTimeDomainArray ----------------- NTTimeDomainArray expresses an array of values that have a constant time between each value. structure NTTimeDomainArray typeIdentifier_t typeIdentifier alarm_t alarm timeStamp_t timeStamp // Time of the first sample [] values double deltaTime // Time in seconds between each sample. NTHistogram ----------- structure NTHistogram typeIdentifier_t typeIdentifier alarm_t alarm timeStamp_t timeStamp double firstValue double lastValue enum_t mode // LINEAR and LOGARITHMIC long Nbuckets // array size long[] values NTNDArray --------- NTFrequencyDomainArray ---------------------- structure NTFrequencyDomainArray typeIdentifier_t typeIdentifier alarm_t alarm timeStamp_t timeStamp // Time of the first sample [] values double deltaFrequency // Frequency in Hz. double firstFrequency // frequency of the value[0]. NTStatistic ----------- NTStatistic is a type to compress time domain data. It may be used for archive data, or compressing time domain data, or time domain rebasing. For instance, you have an FPGA sending data at 10KHz, and you want to archive its output, but you can't archive at the native rate. structure NTStatistic typeIdentifier_t typeIdentifier alarm_t alarm // May be worst alarm from the N samples, or alarm of the aggregation. Which is not defined. timeStamp_t timeStamp // Nominally, the time of aggregation. double firstValue timeStamp_t firstValueTimeStamp double lastValue timeStamp_t lastValueTimeStamp double highValue double lowValue double meanValue double stdDev long N NTImage ------- NTTable ------- NTTable is intended to be used when exchanging data which looks like a table. For instance table of BPM values of a whole accelerator section, or contents of a JDBC ResultSet. structure NTTable typeIdentifier_t typeIdentifier alarm_t alarm timeStamp_t timeStamp string[] labels [] values NTChannelFinderDirectory ------------------------ NTChannelFinderDirectory can be used for exchanges of ChannelFinder query results. structure NTChannelFinderDirectory typeIdentifier_t typeIdentifier alarm_t alarm timeStamp_t timeStamp string[] names double[] positions nameValuePair_t[][] properties // A 2d array of properties (ie tags). Slow moving (first) index corresponds to names, // fast moving (second) index corresponds to the tags of the name. NTXAlarm -------- - to be defined Non-Normative Types =================== Users and developers using EPICS V4 are free to defined their own special purpose types on top of PVStructure (or the other PV Types) but should note that only the normative types deifned here (NT*) above, are guaranteed to be understood by EPICS V4 services and clients.
