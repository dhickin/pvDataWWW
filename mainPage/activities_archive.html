Activities of the EPICS V4 Working Group

Activities of the EPICS Version 4 WG by, archived by month.

7-Nov-2011 to 20-Dec-2011

Concentrating on simplifying EPICS V4 for the user programmer and regularizing APIs. Specifically, present activities include:

  1. Review of ezpva and "gather" continue
  2. ezpva rename. Question of whether to rename, and if so to do it. Depends on whether ezpva will include async acquisition
  3. Implementation of ezpva specification (Marty, Matej)
  4. NT type definitions specifications (Bob)
  5. NT type id definition (requires lead)
  6. NT type implementations (requires lead)
  7. Question of how to encode NTTable; in CORBA Any style, or as variant union
  8. Unsigned Int handling. Document, republish to observers, and implement
  9. Comments on pvAccess specification. Integration of remarks from Michael and others. Edits for lingo. (Matej, Marty, Greg).
  10. pvAccess publication to techtalk
  11. Reorganisation of pvData & pvIOC javadoc into "classical" javadoc plus reference doc (Marty)
  12. Example and tests of use of pvAccess for async 'monitoring'. (Paul Gibbons, Marty, Bob). Outcome should be 1) example code in pvData, 2) Text for a getting started document
  13. Twiss model service design (Guobao, Greg)
  14. Model simulation service (aka "Online Calculator") design (Guobao, Greg)
  15. EPICS V4 getting started developer's guide document preparation (Greg, Matej)
  16. Publishing javadoc for EPICS V4 modules through Maven (Matej, Gabriele)
  17. Logging and setpoint save tool, design and implementation (Guobao, Marty)
  18. Moving meeting hosting service from Adobe Connect to something else (Greg)
  19. Development, code, and documentation standards (Matej, Greg)
  20. use of Doxygen for documentation generation. What standards must be in place so you get useful output, bit just pretty pages with nothing in them (needs lead)
  21. Question: Sending pvStructure to a service. Is it supported? How do you do it? Need demo
  22. Sending pvStructure that changes call to call through ezpva. Requires code changes (Matej, Marty)
  23. Question: Build environment for EPICS V4 in a deployed lab setting (ie, Maven or just Make?)
  24. Deploying rdbService at PSI (Greg)

Oct 2011 to 7-Nov-2011

  1. Normative types. Definitions and binding implementations of the so called, normative types. These are the data types that it is expected all general purpose clients will understand, and it is encouraged that services and IOC applications should supply.
  2. First services, writing and deployment. In particular, a general relational database access service, and lattice modelling services are being actively developed
  3. Display tools and frameworks. pvManager, and CSS plugins
  4. Directory service support, in particular channelFinder, and associated CSS plugins
  5. The Gather Service Framework. This is a server side distributed data acquisition framework, which helps a programmer to construct a service that collects data from a number of sources (EPICS v4 or EPICS V4 records), and makes it available in the normative types suiatble for displaying data of a number of devices. For instance, useful for BPM orbit, or linac energy management services.
  6. Fully implementing the the pvAccess specification working draft.
